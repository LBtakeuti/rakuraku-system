## 1. このプロジェクトは何か

**らくらく受発注システム** — トン屋(問屋)向けの受発注管理システム。株式会社プロスパの現行システムを置き換える後継システムとして新規開発する。

最重要の特徴は **「ITリテラシーが低いユーザーが、マニュアルなしで使えること」**。この一点が他のあらゆる判断に優先する。技術的にエレガントでも、現場の事務員が迷う UI は不採用とする。

## 5. UX設計の鉄則(最重要)

このシステムの成否はUXで決まる。以下は例外なく守ること。

### 5.1 言葉づかい
- **業務動詞でラベルを付ける。** 「受注」ではなく「注文を受ける」、「入荷登録」ではなく「入荷を登録する」。
- 専門用語・英語を画面に出さない。日本語ラベルのみ。
- エラーメッセージは具体例を添える。「入力してください」ではなく「電話番号を入力してください。例: 03-1234-5678」。

### 5.2 画面構成
- **すべての画面の上部に「ホームに戻る」ボタンを置く。**
- 必須項目は赤いバッジ「必須」、任意項目はグレーのバッジ「任意」を必ず表示。
- 複雑なフォームは3ステップのウィザードに分割する(お客様マスターが実例)。
- 削除・送信・発行など取り消せない操作の前には必ず確認ダイアログを出す。
- 削除はできるだけ「元に戻す」(Undo)を用意する。

### 5.3 見た目
- **派手な装飾は禁止。** 影は1層のみ、ホバー時に少し浮く程度。グラデーション・ネオン的配色は使わない。
- アニメーションはやわらかい操作感(250〜350msのイージング)。ただし過剰な演出はしない。
- メインカラーは `#1d4ed8`(青)。
- システム名は「らくらく受発注システム」、ロゴは「楽」の一文字。

### 5.4 モックアップの扱い
`mockups/` の13画面のHTMLは、レイアウト・ラベル文言・色・操作感の**正**である。実装時はこれを忠実に再現すること。CSSの変数名・色コードもモックアップに揃える。勝手に「改善」しない。

### 5.5 コンポーネント使い分け

#### ボタンの実装方針
**モックアップ再現が最優先（§5.4 と整合）**。装飾を伴うボタン（モック独自CSSを使うもの）は素の `<button>` + Tailwind でモックを忠実再現すること。shadcn/ui の `<Button />` は、装飾不要・モック非依存の汎用部品が必要になった時点で導入する。

| 用途 | 採用方針 | 例 |
|------|---------|-----|
| ホームの大きなカード | `<BigButton />`(`src/components/common/`) | 注文を受ける／納品する |
| モック準拠の装飾ボタン（保存・キャンセル・big-btn primary 等） | 素の `<button>` + Tailwind 直書き | フォームの「保存」「キャンセル」、ウィザードの「次へ」 |
| 絞り込みチップ・ステータス選択 | 素の `<button>` + Tailwind 直書き or 専用 Chip コンポーネント | お客様一覧の絞り込みチップ |
| 装飾不要・汎用的なボタン | shadcn/ui `<Button />` | 確認ダイアログ内、エラーステートでの「再試行」等 |

判断基準：**モックに該当ボタンが描かれている → 素 button + Tailwind / 描かれていない汎用UI → shadcn Button**。

#### フォームライブラリ
- マスター画面（顧客・商品）など複雑度が低いフォームは `useState` + Zod `safeParse` + Server Action（`useActionState`）で実装してよい
- 受注画面など**明細行の動的追加が必要なフォーム**は **React Hook Form + `zodResolver` + `useFieldArray`** を使う
- どちらの場合も Zod スキーマは `src/lib/validations/` に集約し、クライアント/サーバー両方で利用する

#### アイコンライブラリ
`lucide-react` のみ使用する。バージョンは現状 `^1.16.0` 系で固定。0.x 系最新への切替は、実害が出た時点で再判断する。

## 6. 業務上の重要な前提

実装の判断に関わる業務ルール。詳細は `docs/01-business-overview.md` と `docs/05-workflows.md`。

- 在庫管理は **引当・実減の二段階方式**。受注時に引当(仮押さえ)、納品時に実減。
- 在庫が不足していても、警告を出すだけで受注は通せる。
- ロット管理は商品ごとに ON/OFF。食品・コスメは ON、雑貨は OFF。
- 在庫引当は FIFO(賞味期限の近い順)で自動。
- 期限警告は60日前に黄色、14日前に赤。
- 倉庫は当面1箇所。ただしテーブルには `warehouse_id` を持たせ、将来の複数倉庫に備える。
- 税率は 10% と 8% の2種類のみ。
- 商品マスターの売上単価は、注文明細の行ごとに上書きできる。
- 納品の確定が売上の計上タイミング。納品書=売上伝票。
- 請求は月次。締め日はお客様ごとに異なる(`customer.closing_day`)。
- 請求書の形式はお客様ごとに異なる(`customer.invoice_format`)。請求書のみ、または請求書+納品書の2段組。

## 7. コーディング規約

- ファイル名は kebab-case(`order-new-page.tsx`)。
- Reactコンポーネント名は PascalCase。
- DBカラム名は snake_case、TypeScript変数は camelCase。変換は型レイヤ(`types/`)で吸収する。
- 金額はすべて number(整数、円単位)で扱い、表示時に通貨フォーマットする。
- 日付は ISO 形式の文字列で持ち、表示時に `YYYY/MM/DD` に整形する。
- Supabaseのクエリは `lib/supabase/` に集約し、画面コンポーネントから直接SQLを書かない。
- バリデーションは Zod スキーマを `lib/validations/` に定義し、フォームとAPIの両方で使う。

## 8. 採番ルール

| 対象 | 形式 | 例 |
|------|------|-----|
| 受注番号 | 8 + 8桁連番 | `800000001` |
| 発注番号 | P + 8桁連番 | `P00000001` |
| 納品書番号 | N + 8桁連番 | `N00000001` |
| 請求書番号 | B + YYYYMM + 連番 | `B202603001` |

採番ロジックは `lib/utils/numbering.ts` に集約する。

## 9. 実装の進め方

`docs/06-implementation-plan.md` にフェーズ分けした実装順序がある。**この順序で進めること。** 各フェーズは前のフェーズの完成を前提にしている。

大原則:
1. まず `supabase/schema.sql` を適用してDBを作る。
2. 共通コンポーネント(TopHeader, PageBar, HomeButton)を先に作る。
3. マスター(お客様・商品)から作る。受発注はマスターに依存する。
4. 受発注 → 在庫 → 納品 → 請求 の順。業務フローの順に作る。
5. 帳票は各画面ができてから対応する出力ボタンを足す。
