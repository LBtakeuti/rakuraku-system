# 処理フロー詳細

主要な業務処理が、データに対して何をどの順で行うかを定義する。各画面の確定ボタンを押したときの裏側の処理を、実装できる粒度で記述する。トランザクション境界も示す。

---

## 全体の流れ

```
注文を受ける
   │  受注を作成、在庫を引当、必要なら発注を自動作成
   ▼
発注 → 入荷を登録する
   │  仕入先から届いた商品を在庫に加算
   ▼
納品する
   │  商品を出荷、在庫を実減、売上(納品書)を計上
   ▼
請求の業務(月次)
   │  納品書を集計して請求書を発行
   ▼
入金を消し込む
      入金と請求を対応づける
```

---

## フロー1: 注文を受ける

画面: `/orders/new`(`mockups/order-new.html`)

### トリガー
「この内容で注文を確定する」ボタン。

### 処理(1トランザクション)

1. **受注ヘッダを作成**
   - `sales_order` に1行 insert。
   - 受注番号を採番(`8` + 8桁連番)。
   - status は `pending`(納品待ち)。
   - お客様、注文日、納品予定日、届け先、メモを保存。

2. **受注明細を作成**
   - 明細行ごとに `sales_order_line` に insert。
   - 商品コード、数量、単価、税率を**スナップショットとして**保存(後で商品マスターが変わっても受注内容は変わらない)。

3. **在庫商品は引当する**
   各明細のうち、商品の `default_order_type = stock` のものについて:
   - FIFO で `product_stock` のロットから引き当てる(賞味期限の近い順、詳細はフロー補足を参照)。
   - 引き当てたロットごとに `sales_order_line_allocation` に insert。
   - `product_stock.quantity_allocated` を加算。
   - `stock_movement` に `allocate` タイプで記録。
   - 引当可能数が足りなくても、警告は画面で出すが処理は止めない(オーバー引当を許容)。

4. **発注が必要な商品は発注を自動作成**
   各明細のうち、商品の `default_order_type = order_at_sale` のものについて:
   - 仕入先ごとに `purchase_order` をまとめて作成(同じ仕入先なら1つの発注書に集約)。
   - 発注番号を採番(`P` + 8桁連番)。
   - `purchase_order_line` に明細を insert。
   - この受注との紐付けを保持(受注伝票・発注書に相互参照を印字するため)。
   - `default_order_type = manual_order` の商品は自動発注しない(担当者が後で手動発注)。

### 完了後
注文一覧に新しい受注が `pending` で表示される。自動作成された発注は発注一覧に表示される。

## フロー2: 入荷を登録する

画面: `/receivings`(`mockups/receivings.html`)

### トリガー
「この内容で入荷を確定する」ボタン。対象の発注書を選んだ状態で実行。

### 処理(1トランザクション)

入荷数量が 0 でない明細について:

1. **入荷先の在庫ロットを決める**
   - ロット管理商品:同じ「仕入先・商品・賞味期限」の `product_stock` 行があれば、それに加算。なければ新規ロット行を作成(ロット番号は入力値、空欄なら `YYYYMMDD-連番3桁` で自動採番)。
   - ロット管理しない商品:その商品の `product_stock` 行(倉庫×商品で1行)に加算。

2. **実在庫を増やす**
   - `product_stock.quantity_on_hand` を入荷数量だけ加算。
   - `stock_movement` に `in` タイプで記録(入荷日、仕入先、発注番号を残す)。

3. **発注の入荷状況を更新**
   - `purchase_order_line.received_quantity` に入荷数量を加算。
   - 全明細が発注数量に達したら `purchase_order.status` を `received`(全量入荷)に、一部なら `partial`(一部入荷)に更新。

### 分納
入荷数量を発注数量より少なく入力した場合、発注は `partial` のまま残り、後日同じ発注書から続きの入荷を登録できる。

## フロー3: 納品する

画面: `/deliveries`(`mockups/delivery.html`)

### トリガー
納品対象の受注を複数選択して「納品を確定する」ボタン。

### 処理(選択された受注ごとに、それぞれ1トランザクション)

1. **納品書(=売上伝票)を作成**
   - `sales_invoice` に1行 insert。
   - 納品書番号を採番(`N` + 8桁連番)。
   - 納品日(=売上計上日)は当日。
   - 元の `sales_order` への参照を保持。

2. **納品明細を作成**
   - `sales_invoice_line` を `sales_order_line` から複製。
   - 数量・単価・税率は受注時のスナップショットを引き継ぐ。

3. **在庫を実減する**
   引き当ててあったロット(`sales_order_line_allocation`)ごとに:
   - `product_stock.quantity_on_hand` を減算(実在庫が物理的に減る)。
   - `product_stock.quantity_allocated` を減算(引当を解除)。
   - `stock_movement` に `out` と `deallocate` の2レコードを記録。
   - この時点で実在庫が足りなければ納品確定できない(エラーで止める)。

4. **受注のステータスを更新**
   - 受注の全明細が納品済みなら `sales_order.status` を `fulfilled`(納品済み)に。
   - 一部のみなら `partial`(一部納品)に。

### 完了後
売上を見る画面に納品書が表示される。これが月次請求の集計対象になる。

### 部分納品について
現状は受注単位で一括納品する想定。明細単位の部分納品は将来対応とし、スキーマは対応できる構造にしておく。

## フロー4: 請求の業務(月次)

画面: `/billing`(`mockups/billing.html`)

### トリガー
ステップ3の「請求書をまとめて発行する」ボタン。

### 前段: 締めと集計(ステップ1〜2)
1. 締め日と対象期間を指定。
2. 対象期間内に納品日がある `sales_invoice` を、お客様ごとに集計。
3. お客様ごとに前回請求額・入金額・繰越額・今回売上・消費税・今回請求額を計算し、一覧表示。
4. この一覧が「請求一覧表」の元データ。

### 発行処理(ステップ3、選択されたお客様ごとに1トランザクション)

1. **請求ヘッダを作成**
   - `billing_statement` に1行 insert。
   - 請求書番号を採番(`B` + YYYYMM + 連番)。
   - 締め日、支払期限、前回請求額、入金額、繰越額、今回請求額を保存。

2. **請求明細を作成**
   - 対象期間の `sales_invoice_line` を集約し、`billing_statement_line` に insert。
   - 税率別(10%・8%)に小計を計算。

3. **帳票PDFを生成**
   - `customer.invoice_format` を見る。
     - `invoice_only` → 請求書(帳票3)のPDFを生成。
     - `invoice_delivery` → 請求書+納品書(帳票4)のPDFを生成。
   - 端数処理は `customer.tax_rounding` に従う。

4. **ステータス更新**
   - `billing_statement.status` を `issued`(請求済み)に。

### 請求一覧表
ステップ2の「請求一覧表を印刷する」で、全対象お客様を1枚にまとめた帳票5を出力する。これは `billing_statement` を作成する前でも、集計結果から出力できる。

## フロー5: 入金を消し込む

### 想定処理
1. お客様から入金があったら `payment` に記録。

## フロー補足: FIFO 引当ロジック

注文を受けるとき、在庫商品をどのロットから引き当てるか。

### 優先順位
1. 賞味期限が近いロットを優先(先に期限が来るものから出す)。
2. 賞味期限が同じなら、入荷日が古いロットを優先。
3. 賞味期限を管理しない商品は、ロットを区別せず1行にまとめて引き当てる。

### 例
商品Aを30個受注。`product_stock` に次の2ロットがある。
- ロットX:賞味期限 2026/05/10、在庫 20個
- ロットY:賞味期限 2026/06/15、在庫 50個

引当の結果:
- ロットXから 20個(期限が近いので優先)
- ロットYから 10個(残り)
- `sales_order_line_allocation` が2行作られる。

### 在庫不足時
引当可能数(実在庫 − 引当中)を超える注文でも、引当はオーバーして記録し、受注は通す。`quantity_allocated` が実在庫を上回る状態を許容する。納品時に実在庫が足りなければ、そこで初めて止まる。

## トランザクションの原則

- 上記の各「処理」は、原則として1つのデータベーストランザクションで実行する。途中で失敗したら全体をロールバックする。
- 採番(受注番号・発注番号など)は、同じトランザクション内で連番の重複が起きないようにする(採番テーブルの行ロック、またはシーケンス利用)。
- 在庫の数量更新は競合に注意する。同じ `product_stock` 行への同時更新は行ロックで直列化する。
